# Architectural Decisions & Technical Choices

- **[Architecture] 3-Layer Memory Architecture (2026-04-22):** Adopté pour persister l'état et optimiser l'utilisation des tokens via le contexte constitutionnel (basé sur l'architecture Jules/Claude-Code).
- **[Architecture] Vanilla Stack Strict (2026-04-15):** Utilisation exclusive de HTML/CSS/JS (Vanilla). Les frameworks de type React ou Tailwind sont proscrits pour garantir la maintenabilité, l'absence d'étapes de build (compilation) et une portabilité "standalone" maximale (compatibilité 100% hors-ligne).
- **[Design] Glassmorphism Design System (2026-04-10):** Choix de l'esthétique Glassmorphism (fonds semi-transparents, backdrop-filters) pour offrir une interface moderne, "premium" et engageante pour les élèves. Toutes les couleurs sont gérées via les variables de `shared.css`.
- **[PWA & Offline] LocalStorage (2026-04-18):** L'API `localStorage` est le mécanisme de persistance unique pour les paramètres (thème, son), les listes d'élèves, et les scores, afin d'éviter la complexité des bases de données locales tout en gardant une fonction hors ligne.
- **[PWA & Offline] Service Worker (2026-04-30):** Le projet intègre un point d'enregistrement unique pour `sw.js` via `assets/js/theme.js`, utilisant une logique agnostique quant aux chemins d'accès. La stratégie "Stale-While-Revalidate" est utilisée pour mettre à jour les assets sans versionnage manuel des caches. *Attention : L'ajout de nouvelles applications nécessite la mise à jour manuelle du tableau `ASSETS` dans `sw.js`.*
- **[Assets] Hébergement Local Strict (2026-04-12):** Aucune dépendance CDN n'est tolérée. Toutes les polices (Outfit, Inter) et les scripts doivent être hébergés localement.
- **[Assets] FontAwesome 7 Pro (2026-04-30):** Remplacement de Lucide par FontAwesome 7 Pro (Uniquement Solid et Duotone). Un script (`meta/scripts/generate_fa_subset.js`) extrait dynamiquement les icônes nécessaires. Ne jamais mettre en cache les nœuds DOM des icônes avant modification (ils sont remplacés par des `<svg>`).
- **[Pédagogie] Adaptive Difficulty (2026-04-25):** Pilotée par le module centralisé `ScoreManager` qui déclenche un événement global `c2_change_difficulty`. Les interfaces des applications doivent écouter cet événement pour mettre à jour la difficulté (plutôt que de s'appuyer sur des compteurs locaux dans chaque vue).
- **[Pédagogie] Contraintes via l'État (2026-04-24):** Les règles pédagogiques (mouvements interdits, obstacles obligatoires) sont appliquées au niveau de la génération de l'état (State), et non de l'interface (UI), assurant la cohérence peu importe la méthode de saisie de l'élève.
- **[Pédagogie] Affichage Binaire (2026-04-25):** Dans les visualisations de calculs binaires, les bits inactifs ("0") doivent toujours être affichés explicitement (ex: `128 + 0 + 32 + 0 = 160`) pour renforcer l'apprentissage des pondérations.
- **[UX] Navigation Top-Tabs (2026-05-01):** Standardisation de la navigation mobile et desktop en utilisant exclusivement des onglets en haut de page (`.tabs`), éliminant les "bottom tab bars" (`.nav-bar` déprécié) pour prévenir les conflits de gestuelles du système d'exploitation mobile.
- **[UX] Contrôles Embarqués (2026-05-01):** Les boutons d'action au sein des éléments de liste dynamique doivent suivre le flux du document (Flexbox relatif) ; l'utilisation de `position: absolute` est proscrite pour éviter les chevauchements inattendus lors de l'expansion.
- **[Sécurité] Atténuation XSS (2026-04-25):** L'usage de la propriété `.innerHTML` est strictement encadré et à éviter pour toute donnée gérée par l'utilisateur. Favoriser `document.createElement`, `textContent` et utiliser `ScoreManager._escapeHtml()`.
- **[Testing & CI/CD] Playwright E2E (2026-04-30):** La vérification du code repose intégralement sur des scripts de test Playwright E2E situés dans `meta/e2e_tests/` (exécutés via GitHub Actions). Il n'existe pas de suites de tests unitaires (ex: pas de pytest/npm test standard) pour le code frontend.
- **[Dev] Propreté du Dépôt :** Interdiction stricte de "commit" des fichiers de travail temporaires, brouillons (`plan.md`, `test_*.py` en dehors du dossier e2e, `server.log`), ou dossiers de cache (`__pycache__`). Les modifications s'opèrent via `write_file` ou des diff Git précis (refus de l'outil `replace_file_content`).
