# Architectural Decisions & Technical Choices

- **[Architecture] 3-Layer Memory Architecture (2026-04-22):** Adopted to persist state and optimize token usage via constitutional context (based on the Jules/Claude-Code architecture).
- **[Architecture] Strict Vanilla Stack (2026-04-15):** Exclusive use of HTML/CSS/JS (Vanilla). Frameworks like React or Tailwind are banned to ensure maintainability, no build steps (compilation), and maximum "standalone" portability (100% offline compatibility).
- **[Design] Glassmorphism Design System (2026-04-10):** Chose the Glassmorphism aesthetic (semi-transparent backgrounds, backdrop-filters) to provide a modern, "premium," and engaging interface for students. All colors are managed via `shared.css` variables.
- **[PWA & Offline] LocalStorage (2026-04-18):** The `localStorage` API is the sole persistence mechanism for settings (theme, sound), student lists, and scores, avoiding local database complexity while maintaining offline functionality.
- **[PWA & Offline] Service Worker (2026-04-30):** The project integrates a single registration point for `sw.js` via `assets/js/theme.js`, using path-agnostic logic. A "Stale-While-Revalidate" strategy is used to update assets without manual cache versioning. *Note: Adding new applications requires manually updating the `ASSETS` array in `sw.js`.*
- **[Assets] Strict Local Hosting (2026-04-12):** No CDN dependencies are tolerated. All fonts (Outfit, Inter) and scripts must be hosted locally.
- **[Assets] FontAwesome 7 Pro (2026-04-30):** Replaced Lucide with FontAwesome 7 Pro (Solid and Duotone only). A script (`meta/scripts/generate_fa_subset.js`) dynamically extracts the necessary icons. Never cache icon DOM nodes before modification (they are replaced by `<svg>` tags).
- **[Pedagogy] Adaptive Difficulty (2026-04-25):** Driven by the centralized `ScoreManager` module, which triggers a global `c2_change_difficulty` event. App interfaces must listen to this event to update difficulty (rather than relying on local counters in each view).
- **[Pedagogy] State-Level Constraints (2026-04-24):** Pedagogical rules (forbidden moves, mandatory obstacles) are enforced at the state generation level, not the UI level, ensuring consistency regardless of the student's input method.
- **[Pedagogy] Binary Display (2026-04-25):** In binary calculation visualizations, inactive bits ("0") must always be explicitly displayed (e.g., `128 + 0 + 32 + 0 = 160`) to reinforce weight learning.
- **[UX] Top-Tabs Navigation (2026-05-01):** Standardization of mobile and desktop navigation using exclusively top tabs (`.tabs`), eliminating bottom tab bars (`.nav-bar` is deprecated) to prevent conflicts with mobile OS gestures.
- **[UX] Embedded Controls (2026-05-01):** Action buttons within dynamic list elements must follow the document flow (relative Flexbox); the use of `position: absolute` is banned to avoid unexpected overlaps during expansion.
- **[Security] XSS Mitigation (2026-04-25):** The use of the `.innerHTML` property is strictly regulated and to be avoided for any user-managed data. Favor `document.createElement`, `textContent`, and use `ScoreManager._escapeHtml()`.
- **[Testing & CI/CD] Playwright E2E (2026-04-30):** Code verification relies entirely on Playwright E2E test scripts located in `meta/e2e_tests/` (run via GitHub Actions). There are no unit test suites (e.g., no standard pytest/npm test) for the frontend code.
- **[Dev] Repository Cleanliness:** Strict ban on committing temporary work files, drafts (`plan.md`, `test_*.py` outside the e2e folder, `server.log`), or cache folders (`__pycache__`). Modifications are made via `write_file` or precise Git diffs (the `replace_file_content` tool is refused).
